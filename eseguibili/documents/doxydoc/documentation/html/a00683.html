Descrizione dettagliata

Classe base astratta per tutti i payload scambiati in rete tra server e client.

Stabilisce il contratto fondamentale del protocollo di rete del progetto. Ogni pacchetto serializzabile viaggiante sui socket deve estendere questa classe. Assicura l'autoconsapevolezza temporale (timestamp) e l'interoperabilità verso la GUI tramite la conversione in CallbackDTO.

FLUSSO DI COMUNICAZIONE CLIENT-SERVER:

  1. Autenticazione: Il client invia una LoginRequest. Il server controlla i dati nel DB e risponde con LoginResponse, specificando anche se l'utente ha privilegi di amministratore.

  2. Registrazione: Se l'utente non ha un account, invia una RegisterRequest. Il server inserisce i dati e risponde con RegisterResponse.

  3. Richiesta Partita: L'utente loggato invia una PlayRequest per cercare un avversario. Il server risponde con PlayResponse, indicando se la partita inizia subito (MATCH_FOUND) o se deve aspettare (WAITING).
  4. Inizio Partita: Appena ci sono due giocatori, il server invia a entrambi un messaggio GameStart contenente la sfida cifrata, il timer e i dati dell'avversario.
  5. Fase di Gioco:
    • Il giocatore elabora la risposta e la invia con AnswerSubmission.
    • Se un giocatore risponde, il server avvisa l'altro in tempo reale tramite OpponentAnswered.

  6. Fine Partita: Il server elabora i tempi e decreta un vincitore, comunicandolo ai due client tramite il messaggio GameResult.

  7. Storico: Dalla lobby, il client può richiedere lo storico delle proprie partite con HistoryRequest. Il server interroga il DB e risponde con HistoryResponse.

  8. Disconnessione:
    • Se un client esce dal gioco regolarmente, invia ClientDisconnect.
    • Se un client cade o si disconnette durante un'attesa o una partita, il server avvisa l'avversario superstite inviando OpponentDisconnected.
